# Архитектура платформы AI Quest Platform

В этом документе описывается архитектурный дизайн, основные принципы проектирования и взаимодействия компонентов в рамках системы **AI Quest Platform**.

---

## 🏛️ 1. Архитектурные принципы

Проект спроектирован с соблюдением принципов разделения ответственности (**Separation of Concerns**), слабой связанности компонентов и высокой модульности:

*   **SOLID**: Каждый класс и сервис имеет единственную ответственность. Взаимодействие между модулями происходит через строго типизированные интерфейсы.
*   **Domain-Driven Design (DDD)**: Платформа разделена на изолированные доменные границы (Bounded Contexts), представленные автономными пакетами в папке `/src/packages`.
*   **Clean Architecture (Чистая Архитектура)**: Ядро бизнес-логики (Quest Engine, AI Engine, Vision Engine, Realtime Engine) ничего не знает о внешнем транспорте (REST API, Express, Socket.io) или конкретных базах данных. Они принимают входные данные через параметры функций и возвращают типизированные структуры.
*   **Repository Pattern**: Все операции с базой данных абстрагированы через слой репозиториев (`DBRepository`), что защищает бизнес-логику от прямых изменений схем БД и позволяет легко переключаться с файловой/In-Memory БД на Prisma PostgreSQL.

---

## 📂 2. Модульная структура пакетов

Вся основная логика вынесена в изолированные NPM-пакеты внутри `/src/packages/`:

```bash
src/packages/
├── types/              # Общие типы данных и структуры сущностей
├── quest-engine/       # Конечный автомат (State Machine) квестов, ранги, XP, инвентарь
├── ai-engine/          # Контекстный генератор, prompt-менеджер, интеграция с Gemini SDK
├── vision-engine/      # Компьютерное зрение (OCR, сканирование QR, проверка схожести, кэширование)
└── realtime-engine/    # Socket.IO шлюз, доставка событий с гарантией ACK, статус присутствия (Presence)
```

### 1. Types Module (`src/packages/types`)
Определяет единые структуры данных для всей платформы:
*   `User`, `Role` (PLAYER, ORGANIZER, ADMIN)
*   `Project` (идентификаторы, названия, шаги квеста, метаданные)
*   `Step` (типы шагов: TEXT, LOCATION, QR, PHOTO, AI_PROMPT)
*   `UserState` (активный шаг, инвентарь, XP, уровень, посещенные места)

### 2. Quest Engine (`src/packages/quest-engine`)
Ядро, отвечающее за прогресс игрока:
*   **State Machine**: Управляет переходами между шагами квеста. Переход на следующий шаг возможен только при выполнении условий валидации (`conditions`).
*   **XP & Ranks Service**: Рассчитывает получаемый опыт, начисляет уровни и выдает игровые звания (от «Новичка» до «Хранителя Тайн»).
*   **Achievement & Inventory Managers**: Контролируют разблокировку достижений и добавление предметов в инвентарь игрока.
*   **Rollback & Checkpoints**: Сохраняет промежуточные состояния игры (чекпоинты), позволяя откатить прогресс в случае спорных игровых ситуаций.

### 3. AI Engine (`src/packages/ai-engine`)
Интеллектуальный ассистент квестов:
*   **Context Builder**: Формирует контекст для модели на основе текущего состояния игрока, его инвентаря, истории чата и описания квеста.
*   **Provider Switching & Fallbacks**: В случае сбоя внешнего API (превышение лимитов Gemini API) плавно переключается на резервный локальный обработчик, чтобы игровой процесс не прерывался.
*   **RAG & Embeddings**: Позволяет искать информацию по лору квеста в загруженных документах для ответов на сложные вопросы.

### 4. Vision Engine (`src/packages/vision-engine`)
Анализ изображений и верификация:
*   **OCR**: Распознавание текста на фотографиях игроков для проверки кодовых фраз (например, надписей на памятниках).
*   **Similarity Engine**: Сравнение загруженного игроком фото с эталоном из базы данных (с вычислением процента схожести).
*   **QR Scanner**: Извлечение закодированных данных и проверка криптографических подписей.
*   **Pipeline & Cache**: Конвейерная обработка снимков с предварительным сжатием и сохранением результатов в кэше для снижения нагрузки на GPU.

### 5. Realtime Engine (`src/packages/realtime-engine`)
Взаимодействие в реальном времени:
*   **Gateway**: Шлюз Socket.IO, обрабатывающий подключения, авторизующий JWT и распределяющий игроков по комнатам (проектам, командам, ролям).
*   **Guaranteed Delivery (ACK & Retries)**: Каждое сообщение снабжается UUID и последовательным номером. Если клиент не подтверждает доставку (ACK), срабатывает очередь повторных отправок с экспоненциальной задержкой.
*   **Synchronization**: При переподключении запрашивает пропущенные события по `Last Event ID`.
*   **Presence Service**: Отслеживает онлайн-статус, качество соединения и активность игроков, отображая данные на панели организатора.

## 🔌 3. Архитектура базы данных

Платформа поддерживает двухслойный доступ к данным:
1.  **Слой разработки/быстрого тестирования**: Локальный JSON-репозиторий (`data_db.json`), поддерживающий синхронную работу без развертывания СУБД.
2.  **Слой Production**: База данных PostgreSQL, управляемая через ORM **Prisma** с поддержкой каскадного удаления, внешних ключей и индексов на часто запрашиваемых полях (`userId`, `projectId`, `stepId`).

Все сущности и связи спроектированы в соответствии с третьей нормальной формой (3NF), обеспечивая высокую целостность и производительность при выполнении сложных транзакций.
