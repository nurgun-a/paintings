# Руководство для разработчиков (Contributing Guide)

Добро пожаловать в команду разработки **AI Quest Platform**! Этот документ устанавливает стандарты кода, правила контроля качества и порядок внесения изменений в проект.

---

## 🛠️ 1. Стандарты написания кода (Code Quality Standards)

Все разработчики обязаны строго следовать правилам статического анализа, настроенным в проекте:

*   **TypeScript**:
    *   Все файлы должны быть написаны исключительно на TypeScript. Использование типа `any` строго запрещено. Используйте точные типы или обобщения (`generics`).
    *   Все перечисления (Enums) должны быть стандартными (`enum`). Использование `const enum` не допускается для обеспечения совместимости модулей.
    *   Импорты типов должны размещаться в верхней части файлов и использовать явные именованные импорты.
*   **Стилизация (Tailwind CSS)**:
    *   Все стили пишутся через утилитарные классы Tailwind непосредственно в разметке компонентов.
    *   Создание дополнительных файлов `.css` или использование CSS-в-JS библиотек запрещено для сохранения высокой скорости отрисовки и малого размера бандла.
*   **Идентификация элементов (HTML ID)**:
    *   Каждый значимый интерактивный элемент (карточки, кнопки, формы, поля ввода, заголовки секций) должен содержать уникальный атрибут `id`. Это необходимо для автоматизированного E2E тестирования (Playwright) и аналитики пользовательского опыта.

---

## 🚦 2. Локальная валидация изменений (CI/CD Gates)

Перед тем как отправить Pull Request, каждый разработчик обязан выполнить полную проверку кода на своем локальном компьютере:

### 1. Запуск статического анализа (Linter & Type Check)
Проверяет синтаксис, неиспользуемые импорты, переменные и соответствие типам:
```bash
npm run lint
```
*Этот шаг должен проходить со 100% успешным результатом (без предупреждений и ошибок).*

### 2. Сборка продакшн-версии приложения (Production Build)
Убедитесь, что оптимизированный сборщик успешно компилирует клиентскую и серверную части:
```bash
npm run build
```

---

## 🧪 3. Требования к покрытию тестами

Каждая новая фича или изменение в логике движков квеста должны сопровождаться написанием тестов в директории `/src/backend/tests`:

*   **Unit-тесты**: Покрывают изолированные методы вычисления опыта, переходов состояний или форматирования контекста.
*   **Integration-тесты**: Проверяют цепочки взаимодействий, например, успешная регистрация пользователя -> инициализация квеста -> отправка ответа -> получение награды -> обновление записи в БД.
*   **Интерактивные проверки**: Запускаются на реальном сервере перед деплоем в Production.
