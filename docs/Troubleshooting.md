# Руководство по устранению неполадок (Troubleshooting & Diagnostics)

В данном руководстве описываются типичные аварийные ситуации, методы их диагностики и пошаговые инструкции по восстановлению работоспособности **AI Quest Platform**.

---

## 🚨 1. Оперативное восстановление при сбое базы данных

Если файлы СУБД повреждены или случайно удалены, запустите процедуру аварийного восстановления из последней резервной копии.

### Шаг 1: Подготовьте резервные архивы
Убедитесь, что файлы резервной копии (базы данных и S3 хранилища) находятся в доступной директории, например `/var/backups/quest_platform/`.

### Шаг 2: Запустите восстановительный скрипт
Передайте пути к бэкапам в качестве аргументов:
```bash
chmod +x deployment/backup/restore.sh
./deployment/backup/restore.sh /var/backups/quest_platform/postgres_dump_20260709.sql.gz /var/backups/quest_platform/minio_assets_20260709.tar.gz
```

## 📈 2. Диагностика метрик и триггеры алертов

Система мониторинга Prometheus непрерывно собирает данные с экспортеров. Ниже описаны основные аварийные триггеры (определенные в `/deployment/monitoring/alerts.yml`) и способы реагирования на них.

### 1. Алерт `ServiceDown` (Критический)
*   **Симптом**: Один из контейнеров ушел в оффлайн.
*   **Действие**:
    Проверьте статус контейнеров через докер:
    ```bash
    docker compose -f deployment/compose/docker-compose.yml ps
    ```
    Проанализируйте логи упавшего контейнера:
    ```bash
    docker compose -f deployment/compose/docker-compose.yml logs -f app
    ```

### 2. Алерт `HighCPUUsage` (Предупреждение)
*   **Симптом**: Загрузка процессора на сервере превышает 85%.
*   **Действие**:
    Чаще всего это связано с высокой нагрузкой на **Vision Engine** при одновременной загрузке сотен фото.
    *   Проверьте системные процессы с помощью `top` или `htop`.
    *   Убедитесь, что кэширование результатов верификации изображений включено и работает исправно.

### 3. Алерт `PostgreSQLConnectionFailed` (Критический)
*   **Симптом**: База данных недоступна для приложения.
*   **Действие**:
    Убедитесь, что процесс Postgres запущен:
    ```bash
    docker compose -f deployment/compose/docker-compose.yml logs postgres
    ```
    Проверьте дисковое пространство. Заполнение диска на 100% — наиболее частая причина аварийной остановки PostgreSQL.

## 🔌 4. Распространенные ошибки подключения к WebSockets

Если игроки жалуются, что они не получают обновления в чате или их статус присутствия не обновляется:

1.  **Ошибка `failed to connect to websocket`**:
    *   *Причина*: На сервере включена балансировка нагрузки без поддержки `sticky sessions`.
    *   *Решение*: Убедитесь, что в Ingress-контроллере включена аннотация `affinity: "cookie"` (см. `/docs/Deployment.md`).
2.  **Пропадание сообщений при обрыве сети**:
    *   *Причина*: Сбой очереди доставки ACK.
    *   *Решение*: Клиент должен автоматически восстанавливать связь с передачей заголовка `lastEventId`. Проверьте, что в логах бэкапа нет ошибок обработки очереди сообщений Redis.
