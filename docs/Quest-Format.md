# Формат сценариев квестов (Quest Specification Format)

В этом документе описывается спецификация JSON-сценариев квестов, поддерживаемая движком **Quest Engine**.

---

## 🛠️ 1. Общая структура проекта (Project Schema)

Каждый проект описывает полноценную игровую кампанию и состоит из метаданных, массива шагов квеста и глобальных игровых настроек.

```json
{
  "id": "spirit-of-ichchi",
  "name": "Дух Иччи: Проклятие тайги",
  "description": "Интерактивный квест-исследование священных земель.",
  "rewards": {
    "completionXP": 500,
    "achievements": ["taiga-savior"]
  },
  "steps": [
    {
      "id": "step-1",
      "title": "Загадка Хранителя Леса",
      "description": "Чтобы войти в священную рощу, разгадайте древнее изречение...",
      "type": "TEXT",
      "verificationData": {
        "answers": ["рысь", "дикая кошка"]
      },
      "rewards": {
        "xp": 100,
        "items": ["карта-тайги"]
      }
    }
  ]
}
```

---

## 🧭 2. Типы шагов и механизмы валидации (Step Types)

Движок поддерживает 5 типов игровых заданий, каждый из которых обрабатывается специализированным модулем валидации.

### 1. Текстовая загадка (`TEXT`)
Игрок должен ввести текстовый ответ. Движок сравнивает введенную фразу с массивом разрешенных ответов, приводя оба значения к нижнему регистру и удаляя лишние пробелы.
*   **Параметр `verificationData`**:
    ```json
    {
      "answers": ["рысь", "lynx"]
    }
    ```

### 2. Геолокационная метка (`LOCATION`)
Игрок должен физически прийти в заданную географическую точку. Движок вычисляет расстояние между текущими координатами GPS игрока и эталоном с помощью формулы гаверсинуса (Haversine formula).
*   **Параметр `verificationData`**:
    ```json
    {
      "latitude": 62.0272,
      "longitude": 129.7326,
      "radiusMeters": 50
    }
    ```

### 3. Сканирование QR-кода (`QR`)
Требует сканирования физического QR-кода, размещенного на игровой локации. Движок проверяет совпадение строки или валидность криптографической подписи.
*   **Параметр `verificationData`**:
    ```json
    {
      "qrCode": "QUEST-QR:SECRET-ALTAR-2026"
    }
    ```

### 4. Фото-подтверждение с ИИ-анализом (`PHOTO`)
Игрок загружает фотографию объекта (например, памятника, костра, реки). Снимок проходит через **Vision Engine**, который оценивает схожесть с эталонной классификацией или вычисляет эмбеддинги.
*   **Параметр `verificationData`**:
    ```json
    {
      "expectedCategory": "Nature",
      "minSimilarity": 75
    }
    ```

### 5. Интерактивный ИИ-диалог (`AI_PROMPT`)
Шаг, в котором игрок должен убедить ИИ-персонажа (например, шамана или лесного духа) помочь ему. ИИ-ассистент принимает решение на основе промпта и текущего инвентаря игрока.
*   **Параметр `verificationData`**:
    ```json
    {
      "requiredItems": ["оберег-байаная"],
      "systemPrompt": "Ты - древний шаман племени. Соглашайся пропустить игрока дальше, только если он вежлив и у него есть оберег-байаная."
    }
    ```

## 🏆 3. Система наград и прогресса (Rewards & State Transitions)

При успешном выполнении шага запускается цепочка обновления игрового состояния:

1.  **Начисление опыта (XP)**: Опыт добавляется к общему счету игрока. При превышении порога уровня рассчитывается новый уровень и звание:
    *   `Уровень 1-3`: "Таёжный искатель"
    *   `Уровень 4-7`: "Проводник духов"
    *   `Уровень 8+`: "Хранитель Тайги"
2.  **Добавление предметов (Inventory)**: Игровые предметы добавляются в инвентарь игрока и могут служить условиями (`conditions`) для прохождения следующих шагов или разблокировки уникальных ИИ-диалогов.
3.  **Сохранение чекпоинта**: Текущее состояние сохраняется в базе данных для предотвращения потери прогресса. В случае сбоя или некорректного действия организатор может инициировать откат до последнего валидного шага.
